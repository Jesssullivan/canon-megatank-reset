# G6020 — Who/What `DeviceIoControl 0x220034` Targets in WICReset (printerpotty.exe)

**Lane B deliverable.** Tell Lane A which driver/handle to bind so that the
extended IEEE-1284 read (IOCTL `0x220034`) returns data on the `04a9:12fe`
service-mode MegaTank. Answer is anchored on a fresh static RE of
`printerpotty.exe` (the WICReset engine) in the reused full Ghidra project
`wicreset-pp-full`.

> Evidence base: PyGhidra over `.ghidra-work/project-full/wicreset-pp-full`
> (read-only), program `printerpotty.exe`. Scripts:
> `.ghidra-work/ioctl_scan.py`, `.ghidra-work/decomp_fns.py`,
> `.ghidra-work/str_refs.py`. Raw dumps: `/tmp/pp-ioctl-scan.txt`,
> `/tmp/pp-220034-fns.txt`, `/tmp/pp-1284parse.txt`.

---

## TL;DR (evidence-final)

1. **`0x220034` is `IOCTL_USBPRINT_GET_1284_ID`, serviced by `usbprint.sys`
   itself** — NOT a usbscan/still-image IOCTL. CTL_CODE decode:
   `DeviceType=0x22 (FILE_DEVICE_USB)`, `Function=0x0d`, `METHOD_BUFFERED`,
   `FILE_ANY_ACCESS`. printerpotty's own symbol for the call site is
   `USBPipe::do_read_1284ID`.

2. **It is issued to the SAME `usb#…{28d78fad-…}` handle WICReset opens**, i.e.
   the `GUID_DEVINTERFACE_USBPRINT` device-interface path passed to `CreateFileW`.
   There is **no separate `\\.\Usbscan%d` handle** anywhere in printerpotty — the
   `Usbscan`/`StiUsb`/`WIA`/`Usb%d` strings simply **do not exist** in the
   binary. The `\\.\Usbscan%d` form belongs to the *other* tool (Canon
   ServiceTool v5103) and to normal-mode WIA usbscan; printerpotty is a pure
   usbprint-class client.

3. **The buffer it expects back is the standard 1284 device-ID string with a
   2-byte big-endian length prefix**: `[len_hi][len_lo]"MFG:Canon;CMD:…;MDL:…;…"`.
   Both 0x220034 call sites strip the first 2 bytes and parse `MFG`/`MDL` keys.

4. **Reconciliation:** `usbprint.sys` services `0x220034` natively. No Canon
   usbscan filter or co-installer is needed for the 1284-ID read. The only
   requirement is that the device be **bound to `usbprint.sys`** (so it exposes a
   `GUID_DEVINTERFACE_USBPRINT` node) and that it answer `GET_DEVICE_ID`. Since
   the `04a9:12fe` service-mode device is a **single USB-printer-class interface**
   (bInterfaceClass 0x07), `usbprint.sys` *will* attach and *will* service
   `0x220034` — provided the VM/passthrough lets `usbprint.sys` claim it. The
   risk is not "wrong driver family"; it is "generic/no driver bound, or wrong
   handle passed."

## 1. The two `0x220034` call sites (from `ioctl_scan.py`)

`DeviceIoControl` call-site sweep, with the IOCTL push captured by walking back
from each call:

```
call@0052e35a fn=FUN_0052dae0   IOCTL 0x220034 @ 0052e345 : PUSH 0x220034   (SetupDi enumeration / discovery)
call@0052d314 fn=FUN_0052d110   IOCTL 0x220034 @ 0052d30c : PUSH 0x220034   (USBPipe::do_read_1284ID)
call@0052ccb4 fn=FUN_0052cab0   IOCTL 0x22003c                              (USBPipe::do_read_vendor   — RECV)
call@0052cfd5 fn=FUN_0052ce40   IOCTL 0x220038                              (USBPipe::do_send_vendor   — SEND)
call@0052d6f4 fn=FUN_0052d4f0   IOCTL 0x220030                              (USBPipe::do_read_status)
```

The USBPipe vtable symbols (from the `USBPipe::*` string xrefs) map the whole
family one-to-one onto the documented `usbprint.sys` IOCTL set:

| Function | IOCTL | usbprint.sys name (FILE_DEVICE_USB=0x22) |
|---|---|---|
| `FUN_0052d4f0` `USBPipe::do_read_status`  | `0x220030` (fn 0x0c) | `IOCTL_USBPRINT_GET_LPT_STATUS` |
| `FUN_0052d110` `USBPipe::do_read_1284ID`  | **`0x220034`** (fn 0x0d) | **`IOCTL_USBPRINT_GET_1284_ID`** |
| `FUN_0052ce40` `USBPipe::do_send_vendor`  | `0x220038` (fn 0x0e) | `IOCTL_USBPRINT_VENDOR_SET_COMMAND` |
| `FUN_0052cab0` `USBPipe::do_read_vendor`  | `0x22003c` (fn 0x0f) | `IOCTL_USBPRINT_VENDOR_GET_COMMAND` |

CTL_CODE = `(DeviceType<<16)|(Access<<14)|(Function<<2)|Method`:
`0x220034` → `DeviceType=0x0022, Function=0x00d, Method=0 (BUFFERED), Access=0`.

## 2. Site A — `USBPipe::do_read_1284ID` (`FUN_0052d110`): same handle, length-prefixed read

The handle comes from `*(param_1 + 0x24)` — the *USBPipe instance's own device
handle*, lazily opened by the shared `do_open` path with the **same**
`CreateFileW(local_104, 0xC0000000, 3, …, OPEN_EXISTING, 0xA0000000, 0)` used by
`do_send_vendor`/`do_read_vendor`. `local_104` is the device-interface path
string (the `usb#vid_04a9&pid_…#…{28d78fad-5a20-4f88-b2b8-a4f5e8a39c2f}` form).

```c
BVar4 = DeviceIoControl(*(HANDLE *)(param_1 + 0x24), 0x220034,
                        lpInBuffer, local_ec,        // in: optional, usually empty
                        local_f8,   5000,            // out buffer (5000 bytes)
                        &local_fc,  (LPOVERLAPPED)0x0);
...
// success path strips the 2-byte 1284 length prefix:
uVar6 = 2;
if (uVar6 < uVar1) {                       // uVar1 = bytes returned
    iVar5 = uVar1 - uVar6;                 // drop first 2 bytes
    param_2[2] = iVar5;
    FUN_008039a0(*param_2, *param_2 + uVar6, iVar5);   // memmove(buf, buf+2, len-2)
}
```

Error strings in this function (`str_refs.py`): `"USBPipe::do_read_1284ID"`,
and the surrounding diagnostic literals `"1284 ID could not be read."`,
`"1284 ID could not be parsed."`, `"1284 ID from the device."`,
`"1284 ID does not contain MDL key."`. So the **expected reply is the standard
IEEE-1284 device-ID blob** whose first two bytes are the big-endian length and
whose body is the ASCII `MFG:…;CMD:…;MDL:…;…` key/value string.

## 3. Site B — discovery/enumeration (`FUN_0052dae0`): opens its OWN usbprint handle

This is the device-discovery routine. It builds the GUID inline and enumerates
the **USBPRINT** interface class, then opens each device-interface path and reads
the 1284 ID to identify the printer (MFG/MDL match):

```c
local_90.Data1 = 0x28d78fad;                          // GUID_DEVINTERFACE_USBPRINT
DeviceInfoSet = SetupDiGetClassDevsW(&local_90, 0, 0, 0x12);  // PRESENT|DEVICEINTERFACE
... SetupDiEnumDeviceInterfaces / SetupDiGetDeviceInterfaceDetailW ...
local_2b8 = DeviceInterfaceDetailData->DevicePath;    // usb#…#{28d78fad…}
local_298 = CreateFileW(local_2b8, 0xC0000000, 3, 0, OPEN_EXISTING, 0xA0000000, 0);
...
BVar12 = DeviceIoControl(local_298, 0x220034,
                         (LPVOID)0x0, 0,              // NO input
                         _Dst, 0x1000,                // 4096-byte output
                         &local_2dc, 0);
if ((BVar12 != 0) && (local_2dc >= 3)) {
    FUN_00416b90((int)_Dst + 2);                      // parse from _Dst+2 (skip 2-byte len)
    cVar8 = FUN_004e30f0(local_338);                  // parse 1284 dictionary
    ...
    FUN_00422380(&DAT_0098648c);   // "MFG" key lookup
    FUN_00422380(&DAT_00986488);   // "MDL" key lookup
}
```

`DAT_0098648c="MFG"` and `DAT_00986488="MDL"` are referenced ONLY by this
enumeration function and by the 1284 key parser `FUN_004e46a0` (`str_refs.py`).

**Both 0x220034 sites therefore target a `GUID_DEVINTERFACE_USBPRINT`
device-interface handle and parse a 2-byte-length-prefixed 1284 ID.** Site B
opens its own handle during discovery; Site A reuses the persistent USBPipe
handle. Neither uses a still-image / usbscan node.

## 4. Answering the four lane questions

**(1) Same usbprint handle, or a separate Usbscan still-image handle?**
The **same** `usb#…{28d78fad-…}` `GUID_DEVINTERFACE_USBPRINT` handle. printerpotty
contains zero references to `\\.\Usbscan%d`, `usbscan`, `StiUsb`, `WIA`, or
`IOCTL_USBSCAN`. The string `\\.\Usbscan%d` (printf `%d`) is a **ServiceTool
v5103 / normal-mode WIA** artifact, not a WICReset/printerpotty one. WICReset's
entire USB I/O is the `USBPipe` class over usbprint IOCTLs.

**(2) What 1284-ID buffer does it expect back?**
The standard IEEE-1284 device-ID string, **length-prefixed with a 2-byte
big-endian length**, body = ASCII `MFG:Canon;CMD:…;MDL:…;CLS:PRINTER;…`. Both
sites discard the first 2 bytes (`memmove(buf, buf+2, len-2)` at Site A;
`parse(_Dst+2)` at Site B) and then look up the `MFG` and `MDL` keys. A reply
with `len < 3` is treated as failure.

**(3) Can `0x220034` ever succeed on the single-interface `12fe` device?**
Yes — *if* the interface is bound to `usbprint.sys`. `0x220034` is
`FILE_DEVICE_USB` and is serviced by `usbprint.sys`, which is the function driver
for any **USB-Printer-class (bInterfaceClass 0x07) interface**. The `04a9:12fe`
service-mode device is exactly that: a single printer-class interface (EP 0x01
OUT / 0x82 IN), per the repo's own lsusb evidence
(`canon-servicemode-transport-research.md` §0/§2). It has **no still-image
interface**, but `0x220034` does **not need one** — `GET_1284_ID` is a
printer-class operation. `usbprint.sys` fulfils `GET_1284_ID` by issuing the
USB-Printer-class **GET_DEVICE_ID** control request (bRequest 0x00) on EP0; the
device answers from firmware regardless of having a scan interface. So the
single printer-class interface is sufficient. The real-Windows-stack "extra
node" the VM lacks is **not** a usbscan node — it is simply `usbprint.sys`
itself binding the interface and publishing a `GUID_DEVINTERFACE_USBPRINT`
symbolic link. Under a generic/no-driver passthrough (or WinUSB/libusbK bound to
the interface), there is no usbprint device node, so `SetupDiGetClassDevs` finds
nothing and `CreateFile` has nothing to open → `0x220034` can never be issued.

**(4) Does `usbprint.sys` itself service `0x220034`, or is a Canon usbscan
filter/co-installer required?**
`usbprint.sys` services it itself. `IOCTL_USBPRINT_GET_1284_ID` (0x220034),
`GET_LPT_STATUS` (0x220030), and `VENDOR_GET/SET_COMMAND` (0x22003c/0x220038) are
all native `usbprint.sys` IOCTLs in the `FILE_DEVICE_USB` family. No Canon
usbscan filter or co-installer is involved for the 1284-ID read. (Canon's
co-installers matter only for the normal-mode WIA/usbscan still-image path, which
WICReset does not touch.) This reconciles the apparent paradox: WICReset opens
`GUID_DEVINTERFACE_USBPRINT` to **enumerate** *and* issues `0x220034` to the same
handle — because `0x220034` is a usbprint IOCTL, not a usbscan one. There is no
cross-driver hop.

> Repo cross-check: `servicemode-ioctl-0x16000c.md` historically tagged the
> `0x22` family as "usbscan." That labeling is imprecise for printerpotty/
> WICReset: the `0x22` (FILE_DEVICE_USB) IOCTLs here are issued on
> `GUID_DEVINTERFACE_USBPRINT` handles and carry the `usbprint.sys`
> GET_1284_ID/GET_LPT_STATUS/VENDOR_* function codes. Canon ServiceTool v5103 is
> the tool that uses the `\\.\Usbscan%d` device form; the two tools share the
> numeric device-type 0x22 but bind different stacks.

## 5. Recommendation to Lane A (concrete binding)

Bind the service-mode `04a9:12fe` interface to **`usbprint.sys`** so a
`GUID_DEVINTERFACE_USBPRINT` device node exists, then let WICReset enumerate and
read it normally. Concretely, in priority order:

1. **Force the in-box Microsoft `usbprint.sys` (USB printer class) onto the
   `12fe` interface.** Since `12fe` advertises bInterfaceClass 0x07, plugging it
   into a real Windows host (or a VM with proper class-driver matching) auto-binds
   `usbprint.sys` and publishes the interface — `0x220034` then succeeds. If the
   VM passthrough handed the device to WinUSB/libusbK/no-driver, **re-bind it to
   `usbprint.sys`** (Update Driver → "USB Printing Support") or install a
   minimal printer INF that matches `USB\VID_04A9&PID_12FE` to the
   `Printer`/`usbprint` class. This is the lowest-risk fix and matches exactly
   what printerpotty expects.

2. **On Linux (libusb reimplementation), do NOT emulate a usbscan node.**
   Reproduce `0x220034` directly as the USB-Printer-class **GET_DEVICE_ID**
   control transfer on EP0:
   `bmRequestType=0xA1 (class, IN, interface)`, `bRequest=0x00`,
   `wValue=0x0000 (config 0)`, `wIndex=(iface<<8)|altsetting` (typically
   iface 0), `wLength≈1024`. The reply is the 2-byte-length-prefixed
   `MFG:Canon;…;MDL:…` string. That single control transfer is functionally
   identical to what `usbprint.sys` does under `0x220034`; strip the 2-byte
   prefix and match `MFG`/`MDL` just like both decompiled sites do.

3. **Verification gate before trusting the reset path:** confirm
   `GET_DEVICE_ID` returns `MFG:Canon` and an `MDL:` containing the G6020 model
   token. printerpotty bails (`local_2dc < 3` / "does not contain MDL key") if it
   doesn't — so a passing 1284-ID read is the prerequisite for the subsequent
   `0x220038` SEND / `0x22003c` RECV maintenance exchange Lane A is wiring.

**Bottom line for Lane A:** bind `usbprint.sys` (printer class), open the
`GUID_DEVINTERFACE_USBPRINT` device-interface path, and 1284-ID (`0x220034`)
returns data. Do not chase a usbscan/still-image node — printerpotty never uses
one, and the `12fe` device does not expose one.
