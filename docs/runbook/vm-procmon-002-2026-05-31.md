## TL;DR (the answer)

Against the service-mode G6020 (`04a9:12fe`, bound to `usbprint.sys`), a device
function in ServiceTool v5103 raises the modal dialog
**`Error!` / `(error code : 002)`** — OCR-confirmed (dialog class `#32770`, rect
505,326–775,514, body lines "Error!" and "(error code : 002)", OK button; also
screenshotted, `/tmp/dlg002_c3x.jpg`).

****The decisive, perhaps surprising, ProcMon finding: when ServiceTool throws
error 002 it performs NO device I/O of any kind — not even a probe.** Of the 325
ServiceTool rows captured at the click, 152 are real I/O (the rest are ProcMon's
own profiling samples), and of those:

| metric | count |
|---|---|
| `DeviceIoControl` (any) | **0** |
| `CreateFile` on `{28d78fad}` / `\??\USB` / `\Device\USB` | **0** |
| registry reads under `Enum\USB` (any USB device) | **0** |
| rows referencing `12FE`, `04A9`, `usbprint`, `WinUsb`, or `SymbolicName` | **0** |

The ~3-second I/O burst at the click is **entirely** ServiceTool reading and
memory-mapping its own image (`C:\canon\ServiceTool_v5103.exe`) and loading
`imageres.dll` — `CreateFile NAME NOT FOUND C:\canon\imageres.dll` then SUCCESS
on `C:\Windows\SysWOW64\imageres.dll` + `...\SystemResources\imageres.dll.mun`,
i.e. it is fetching the **yellow warning icon for the 002 dialog**. That is the
whole story: load the error icon, show the dialog. **It never touches the printer
device, the WinUSB GUID, the registry `Enum\USB` tree, or the `{28d78fad}`
USBPRINT interface at the time of the error.**

=> **Error 002 = "no usable service-mode maintenance device" decided up front
(cached at app/device-search time), not a failed CreateFile/DeviceIoControl on
the wire.** The RE's `CreateFile \??\USB#…{28d78fad}` + `DeviceIoControl 0x16000c`
handshake is **never reached** in service mode; ServiceTool short-circuits to 002
before issuing any device call.

## Why 002

ServiceTool's device layer is a **WinUSB** consumer (per the prior RE). At
startup / device-search it enumerates USB devices and accepts only a Canon
maintenance interface bound to **WinUSB** (which publishes a device-interface
GUID). In **service mode** the printer presents a single printer-class interface
(`04a9:12fe`) that Windows binds to in-box **`usbprint.sys`** — which registers
GUID_DEVINTERFACE_USBPRINT (`{28d78fad}`) but **no WinUSB interface GUID, and no
Canon INF claims PID_12FE**. ServiceTool therefore concludes there is no
addressable maintenance device and **caches that verdict**; thereafter every
device function (EEPROM, the absorber Clear/Set, etc.) returns error 002
immediately, with no device I/O — exactly what ProcMon shows.

This explains the I/O-level "why" behind the prior runbook
(`wicreset-live-capture-2026-05-31.md`): the official Canon driver binds Canon
WinUSB only to the **normal-mode** MI_04/MI_05 (PID_1865); service mode falls
back to usbprint, so ServiceTool can't bind it and never gets to the
`0x16000c` handshake.

## Can it be made to complete in-VM? (unblocking the usbmon capture)

**Yes — bind the lone 12FE interface to WinUSB so ServiceTool's startup
enumeration finds a device it will open** (runbook Option B):

1. **(Recommended) Force WinUSB onto `USB\VID_04A9&PID_12FE` via a WinUSB INF
   (pnputil) or Zadig.** Service mode = a single interface, so match the bare
   `VID_04A9&PID_12FE` hardware ID (no `&MI_xx`). WinUSB.sys then publishes a
   device-interface GUID; ServiceTool's enumeration should accept the device,
   stop returning 002, and proceed to `CreateFile` the `{28d78fad}`/WinUSB
   interface + issue its `DeviceIoControl 0x16000c` maintenance frames — the
   moment host `usbmon`/`dumpcap` on Bus 001 captures the real absorber reset.
   Needs **no key and no Canon driver**.
   - **Re-test after the override:** confirm the next ProcMon run now shows
     ServiceTool's device `CreateFile` + `DeviceIoControl` (currently both 0).
     The prior runbook cautions the tool's RE'd transport may expect a
     still-image/usbscan interface absent in service mode — resolve empirically;
     but this capture proves the tool isn't attempting ANY device I/O today, so
     the binding fix is the clear, correct next lever.
2. **Official full Canon driver does NOT help** (already verified): no INF
   matches PID_12FE.
3. **A print queue/port is irrelevant** — ProcMon shows zero spooler/port I/O;
   ServiceTool does not go through the print queue. (A `Canon G6000 series` queue
   on `USB001` exists but plays no part.)
4. A **registry-only** WinUSB-GUID tweak would be insufficient — the *functional
   driver* must actually be WinUSB for the subsequent device open/IOCTLs to work.
